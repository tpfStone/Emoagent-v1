## 💡 低优先级决策点

### 14. 用户系统

**当前阶段**：匿名会话（session_id + localStorage）

**结论**：不急，分阶段推进

**发展路径**：
- **阶段一（当前）**：匿名 + 短期记忆（Redis 6轮24h）
- **阶段二（规划中）**：匿名 + 长期记忆（pgvector检索）
- **阶段三（可选）**：用户登录 + 跨设备同步

**技术兼容性**：Session表可添加可选`user_id`字段，支持匿名+登录双模式

**当前存储架构**：
1. **浏览器 localStorage** - 存储session_id + token，永久保存
2. **Redis 短期记忆** - 最近6轮对话，24小时TTL
3. **PostgreSQL 完整历史** - 所有对话记录，永久保存

**实际效果**：
- 同一浏览器永久保持session_id
- AI记住24h内最近6轮对话
- 换设备/清除浏览器 → 失去历史

**优先级**：💡 低（先完成长期记忆，再考虑用户系统）  
**负责人**：后端团队 + 前端团队  
**工作量**：5天

- ✅ 国际化支持（可选）

## 23. LLMOps监控详解

### 为什么需要LLMOps监控？

作为情绪对话系统，LLM是**核心且高成本组件**，需要专门监控：

**四大价值**：
1. **成本可控**：实时追踪token消耗和费用，防止预算超支
2. **质量保证**：监控调用成功率、超时率、降级率
3. **性能优化**：区分不同Provider的延迟和效果
4. **决策支持**：为Provider选择、缓存策略、Prompt优化提供数据

---

### 阶段一：轻量级方案（当前实施）

**实施方式**：在Prometheus中扩展自定义指标

**核心指标**：

| 类别 | 指标名称 | 类型 | 标签 | 说明 |
|------|---------|------|------|------|
| **成本** | `emoagent_llm_tokens_total` | Counter | type, provider | Token消耗（prompt/completion） |
| **成本** | `emoagent_llm_cost_usd` | Counter | provider | 成本估算（美元） |
| **质量** | `emoagent_llm_calls_total` | Counter | provider, status | 调用状态统计 |
| **质量** | `emoagent_llm_retry_total` | Counter | provider | 重试次数 |
| **性能** | `emoagent_llm_latency_seconds` | Histogram | provider, status | 调用延迟 |
| **质量** | `emoagent_llm_response_length` | Histogram | provider | 响应长度 |

**Grafana仪表板面板**：
- Token消耗趋势图（区分prompt/completion）
- 累计成本估算（按天/周/月）
- 调用成功率（success/timeout/error/fallback占比）
- Provider对比（成本、延迟、成功率）
- 异常告警阈值配置

**成本计算示例**（DeepSeek）：

```python
# 定价（需定期更新）
PRICING = {
    'deepseek-chat': {
        'prompt': 0.0014 / 1000,      # $0.0014/1k tokens
        'completion': 0.002 / 1000     # $0.002/1k tokens
    }
}

# 成本计算
cost = (
    prompt_tokens * PRICING[model]['prompt'] +
    completion_tokens * PRICING[model]['completion']
)
```

**优点**：
- 无需额外依赖，集成简单
- 轻量级，不影响性能
- 数据留在本地，隐私安全
- 工作量：+0.5天

**缺点**：
- 功能有限，无完整trace
- 需手动维护定价配置
- 无Prompt版本管理

---

### 阶段二：专业平台（后期可选）

**升级时机**（满足任一条件）：
- 日LLM调用量 > 10,000次
- 需要详细的调用链路追踪（trace）
- 需要Prompt版本管理和A/B测试
- 需要自动化的Prompt优化建议
- 团队规模扩大，需要多人协作

**推荐工具对比**：

| 工具 | 开源 | 自托管 | 特点 | 适用场景 |
|------|------|--------|------|---------|
| **LangSmith** | ❌ | ❌ | LangChain官方，trace完善，prompt hub | 已使用LangChain |
| **LangFuse** | ✅ | ✅ | 开源，隐私友好，功能全面 | 数据敏感场景 |
| **Helicone** | ❌ | ❌ | 专注成本优化，缓存功能强 | 成本压力大 |
| **Weights & Biases** | ❌ | ❌ | ML实验管理，适合模型迭代 | 频繁调优prompt |

**推荐选择**：**LangFuse**（开源、可自托管、功能全面）

**增强功能**：
- 自动追踪每次LLM调用的完整上下文（prompt、response、metadata）
- 可视化trace链路（emotion detection → prompt building → LLM call → response parsing）
- 自动计算成本（支持多Provider）
- Prompt版本管理和A/B测试
- 异常告警和自动化优化建议

**集成示例**（LangFuse）：

```python
from langfuse.decorators import observe

@observe()
async def generate_response(self, prompt: str) -> str:
    # 原有逻辑，自动追踪所有调用
    response = await self.client.chat.completions.create(...)
    return response
```

**迁移成本**：
- 代码改动：1-2天（添加装饰器和配置）
- 学习成本：1天（熟悉平台）
- 数据迁移：不需要（新老系统可并行）

**工作量**：+2天

---

### 实施建议

1. **近期（2周内）**：实施阶段一轻量级方案，快速建立基础监控
2. **中期（2-6周）**：观察数据，调整告警阈值，优化成本
3. **长期（6周后）**：根据业务量和需求，评估是否升级到阶段二

**关键注意事项**：
- 定价需定期更新（建议配置在 `app/config.py` 中）
- 不要在指标中记录用户消息内容（隐私保护）
- 设置合理的告警阈值（如：日成本>$10、错误率>5%）
- 定期review监控数据，优化Prompt和缓存策略
