# 并发控制方案调研与实现

## 一、主流并发控制方案调研对比表

| 方案名称 | 核心实现 | 适配场景 | 核心优点 | 核心缺点 | 项目适配度 | 集成难度 | 部署成本 |
|---------|---------|---------|---------|---------|-----------|---------|---------|
| SlowAPI + Limiter（Redis存储） | 基于FastAPI/Starlette的第三方限速库，支持令牌桶/漏桶/固定窗口，对接Redis做分布式计数 | 生产环境·前端API全局速率限制、LLM接口单用户限速 | 1. 专为FastAPI设计，语法贴合；2. 分布式Redis存储适配多实例Docker部署；3. 支持多维度限速；4. 配置灵活 | 1. 需额外安装依赖；2. 无原生队列能力，仅做限速不处理过载请求 | ★★★★★ | 低（仅需引入依赖+少量配置） | 极低（复用现有Redis，无新增组件） |
| FastAPI原生依赖+Redis令牌桶（自定义） | 基于FastAPI依赖项实现，复用项目Redis7+做令牌桶算法的分布式计数 | 生产环境·前端API/LLM接口精细化限速，需自定义规则 | 1. 完全复用项目现有组件，无第三方库依赖；2. 规则可高度适配儿童情绪分析场景（如单对话轮次限速）；3. 轻量化 | 1. 需手动编写令牌桶核心逻辑，开发量略增；2. 需做异常处理和边界校验 | ★★★★★ | 中（自定义逻辑约50-100行代码） | 无（纯代码实现，复用现有组件） |

| vLLM 批处理调度 | 私有化部署LLM时，基于vLLM的动态批处理/连续批处理特性，调度LLM推理请求 | 生产环境·项目后期私有化部署LLM（如微调CBT儿童情绪模型），替代公有云LLM API | 1. 大幅提升LLM推理吞吐量，降低单请求推理成本；2. 支持动态调整批处理大小，兼顾延迟和并发；3. 适配FastAPI接口调用 | 1. 仅适用于私有化LLM部署，对服务器硬件有要求；2. 与公有云LLM API不兼容 | ★★★☆☆（仅后期私有化场景） | 中（需部署vLLM服务+对接FastAPI） | 中（需GPU服务器，无公有云API调用费） |
| 内存限速（MemoryStorage） | SlowAPI/Starlette Limiter的本地内存计数模式，无外部存储依赖 | 开发/测试环境·单实例调试，临时限速 | 1. 零配置，快速集成；2. 无外部依赖，调试便捷 | 1. 不支持多实例Docker部署；2. 服务重启后计数丢失；3. 生产环境存在并发安全问题 | ★★☆☆☆ | 极低 | 无 |
| Starlette Limiter | Starlette轻量级限速库，FastAPI基于Starlette实现，支持基础限速策略 | 开发/测试环境·轻量限速，或生产环境简单接口限速 | 1. 比SlowAPI更轻量，依赖少；2. 语法简单，快速上手 | 1. 功能比SlowAPI弱，不支持复杂多维度限速；2. 分布式部署需额外对接Redis，文档较简略 | ★★★☆☆ | 低 | 极低（复用Redis时无额外成本） |

### 方案选型建议

结合项目现阶段（Mock→公有云LLM API）和后期（私有化LLM部署）需求：

1. **核心方案**：SlowAPI + Limiter（Redis存储）做前端API全局速率限制，Celery + Redis做LLM请求专用异步队列，兼顾低延迟和成本精准控制，完全复用项目现有Redis组件，适配Docker部署；
2. **备用方案**：若暂不希望新增Celery组件，先采用FastAPI原生依赖+Redis令牌桶做LLM接口精细化限速，后期可平滑升级为Celery队列；
3. **后期扩展**：私有化部署LLM（如CBT微调模型）时，叠加vLLM批处理调度提升推理效率。

## 二、具体实现方案

### 方案一：SlowAPI + Redis

#### 1. 核心代码实现

项目后端为 `app/` 下 Handler-Service-DAO 三层架构，将限速逻辑封装为全局依赖（`app/utils/rate_limit.py`），可在路由层全局/局部引用，适配IP维度和对话ID维度。

**步骤1：创建限速工具类（`app/utils/rate_limit.py`）**

```python
from fastapi import Request, HTTPException
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.errors import RateLimitExceeded
from slowapi.util import get_remote_address
from slowapi.storage import RedisStorage
from slowapi.middleware import SlowAPIMiddleware
import redis
from app.core.config import settings  # 项目现有配置类，复用REDIS_URL

# 复用项目Redis连接，初始化Redis存储
redis_client = redis.from_url(settings.REDIS_URL, decode_responses=True)
storage = RedisStorage(redis_client)

# 初始化限流器，默认按IP限速（项目匿名访问，无用户ID）
limiter = Limiter(key_func=get_remote_address, storage=storage)

# 自定义对话ID维度限速key（贴合项目：单对话轮次防高频请求）
def get_chat_id(request: Request):
    chat_id = request.headers.get("X-Chat-Id") or request.query_params.get("chat_id")
    if not chat_id:
        raise HTTPException(status_code=400, detail="缺少对话ID（X-Chat-Id）")
    return chat_id

# 全局LLM接口限流器（按对话ID，贴合儿童单对话低频次交互）
llm_limiter = Limiter(key_func=get_chat_id, storage=storage)

# 注册限速异常处理器
def register_rate_limit(app):
    app.add_middleware(SlowAPIMiddleware)
    app.state.limiter = limiter
    app.state.llm_limiter = llm_limiter
    app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)
```

**步骤2：在FastAPI主入口注册（`app/main.py`）**

```python
from fastapi import FastAPI
from app.utils.rate_limit import register_rate_limit
from app.handlers import chat, emotion, self_evaluation  # 项目现有路由

app = FastAPI(title="Emoagent-v1 儿童情绪分析系统")

# 注册限速中间件和异常处理器（核心步骤）
register_rate_limit(app)

# 挂载现有路由
app.include_router(chat.router, prefix="/api/chat", tags=["聊天接口"])
app.include_router(emotion.router, prefix="/api/emotion", tags=["情绪识别接口"])
app.include_router(self_evaluation.router, prefix="/api/eval", tags=["自评接口"])

if __name__ == "__main__":
    import uvicorn
    uvicorn.run("app.main:app", reload=True)
```

**步骤3：在路由层添加限速规则（贴合儿童场景，低速率高容错）**

针对前端聊天接口做精细化限速，其他接口做全局IP限速，规则贴合儿童交互习惯，与项目现有LLM 10s超时降级规则对齐。

```python
# app/handlers/chat.py（项目现有聊天路由）
from fastapi import APIRouter, Depends
from slowapi import Limiter, RateLimitExceeded
from app.utils.rate_limit import limiter, llm_limiter
from app.services import chat_service  # 项目现有LLM调用服务

router = APIRouter()

# 核心LLM聊天接口：按对话ID限速（10s/次），按IP限速（60次/分钟），双重控制
@router.post("/send")
@limiter.limit("60/minute")  # 单IP每分钟60次，防止恶意请求
@llm_limiter.limit("1/10seconds")  # 单对话每10秒1次，贴合儿童低频次交互，控制LLM调用成本
async def send_chat_message(chat_req: ChatRequest, limit=Depends(limiter)):
    # 原有业务逻辑，调用LLM服务
    result = await chat_service.handle_chat(chat_req)
    return result

# 其他非LLM接口：全局IP限速（120次/分钟）
@router.get("/history")
@limiter.limit("120/minute")
async def get_chat_history(chat_id: str, limit=Depends(limiter)):
    result = await chat_service.get_history(chat_id)
    return result
```


在 `app/utils/rate_limit.py` 中读取环境变量，实现动态规则。

#### 3. 过载兜底

当速率限制触发时，SlowAPI会抛出 `429 Too Many Requests` 异常，在项目全局错误处理器（`app/utils/exception_handler.py`）中捕获，返回儿童友好的降级话术，与项目现有LLM超时降级逻辑统一：

```python
from fastapi import Request, FastAPI
from fastapi.responses import JSONResponse
from slowapi.errors import RateLimitExceeded

def register_exception_handler(app: FastAPI):
    # 捕获速率限制异常，返回儿童友好话术
    @app.exception_handler(RateLimitExceeded)
    async def rate_limit_exception_handler(request: Request, exc: RateLimitExceeded):
        return JSONResponse(
            status_code=429,
            content={
                "code": 429,
                "msg": "别急呀，我们慢慢聊～ 稍等几秒再和我说话吧😜",  # 儿童友好话术
                "data": None,
                "is_degrade": True  # 标记为降级，方便前端做交互提示
            }
        )
    # 原有LLM超时降级异常处理...
```

### 方案二：进阶版 - Celery + Redis LLM异步请求队列（精准控成本，适配高并发）

适用于项目后期公网部署/高并发场景，在基础版速率限制之上，新增LLM请求异步队列，从根源控制LLM API调用并发数，解决"速率限制仍可能导致短时间内LLM调用量突增"的问题，完全复用项目Redis做broker，适配Docker编排。

#### 1. 核心架构设计

在项目现有 Handler-Service-DAO 三层架构中，新增 Task 层（Celery异步任务），将LLM调用逻辑封装为异步任务，实现：

- 前端请求 → Handler层接收 → 提交LLM任务到Celery队列 → 立即返回任务ID给前端；
- 前端通过任务ID轮询/websocket获取LLM处理结果；
- Celery Worker配置固定并发数，精准控制LLM API同时调用量，从根源控制成本；
- 任务超时时间设为10s，与项目现有LLM超时降级规则对齐，超时则触发降级。

#### 3. 核心代码实现

**步骤1：初始化Celery实例（`app/core/celery.py`，贴合项目配置）**

```python
from celery import Celery
from app.core.config import settings  # 项目现有配置类，复用REDIS_URL

# 初始化Celery，Redis做broker/backend，复用项目REDIS_URL
celery = Celery(
    "emoagent_llm_tasks",
    broker=settings.REDIS_URL,
    backend=settings.REDIS_URL,
    include=["app.tasks.llm_tasks"]  # LLM任务所在目录
)

# 配置任务超时（10s，与项目LLM超时降级规则对齐）
celery.conf.update(
    task_time_limit=10,
    task_soft_time_limit=8,
    task_retries=1,       # 任务失败重试1次
    retry_delay=1,        # 重试延迟1s
    result_expires=3600,  # 任务结果过期时间1小时，减少Redis存储
)
```

**步骤2：封装LLM异步任务（`app/tasks/llm_tasks.py`，核心控成本）**

将项目现有LLM调用逻辑封装为Celery任务，任务并发数由 `docker-compose.yml` 中 `--concurrency` 控制，从根源限制LLM API调用量：

```python
from app.core.celery import celery
from app.services.llm_service import call_llm  # 项目现有LLM调用服务（Mock→DeepSeek）
from app.utils.degrade import llm_degrade  # 项目现有LLM降级工具

# LLM聊天处理任务（核心，异步执行）
@celery.task(bind=True, name="llm_chat_task")
def llm_chat_task(self, chat_id: str, user_input: str, emotion_label: str):
    try:
        # 原有LLM调用逻辑，传入儿童情绪标签（BERT识别结果），贴合儿童情绪分析场景
        llm_response = call_llm(user_input, emotion_label, chat_id)
        return {
            "chat_id": chat_id,
            "response": llm_response,
            "is_degrade": False,
            "msg": "success"
        }
    except Exception as e:
        # 任务超时/失败，触发项目原有LLM降级机制，返回儿童友好话术
        self.retry(exc=e, max_retries=1)
        return {
            "chat_id": chat_id,
            "response": llm_degrade(emotion_label),  # 根据情绪标签返回个性化降级话术
            "is_degrade": True,
            "msg": f"llm_error: {str(e)}"
        }
```

**步骤3：修改聊天接口，提交异步任务（`app/handlers/chat.py`）**

保留基础版速率限制，将同步LLM调用改为异步任务提交，立即返回任务ID给前端，无前端等待延迟：

```python
from fastapi import APIRouter, Depends
from app.utils.rate_limit import limiter, llm_limiter
from app.tasks.llm_tasks import llm_chat_task
from app.schemas.chat import ChatRequest, TaskResponse  # 新增TaskResponse模型

router = APIRouter()

# 核心聊天接口：速率限制+提交LLM异步任务
@router.post("/send", response_model=TaskResponse)
@limiter.limit("60/minute")
@llm_limiter.limit("1/10seconds")
async def send_chat_message(chat_req: ChatRequest, limit=Depends(limiter)):
    # 提交Celery异步任务，立即返回任务ID，无等待
    task = llm_chat_task.delay(
        chat_id=chat_req.chat_id,
        user_input=chat_req.user_input,
        emotion_label=chat_req.emotion_label  # BERT提前识别的情绪标签
    )
    # 返回任务ID给前端，前端轮询/get_task_result获取结果
    return {
        "code": 200,
        "msg": "任务已接收，正在处理～",
        "data": {
            "task_id": task.id,
            "chat_id": chat_req.chat_id
        }
    }

# 新增：根据任务ID获取LLM处理结果（前端轮询接口）
@router.get("/task/result")
@limiter.limit("300/minute")  # 轮询接口放宽限速
async def get_task_result(task_id: str, limit=Depends(limiter)):
    task = llm_chat_task.AsyncResult(task_id)
    if task.ready():
        return {
            "code": 200,
            "msg": "处理完成",
            "data": {
                "task_status": task.status,
                "result": task.result
            }
        }
    else:
        return {
            "code": 202,
            "msg": "正在处理中，请稍等～",
            "data": {
                "task_status": task.status
            }
        }
```

**步骤4：前端轻微适配（无架构修改，仅新增轮询逻辑）**

前端保留原有聊天交互，仅在发送消息后根据任务ID轮询 `/api/chat/task/result` 接口，轮询间隔设为500ms，适配儿童低延迟体验：

```typescript
// 前端聊天发送逻辑（React/TypeScript，贴合项目前端技术栈）
const sendMessage = async (chatId, userInput, emotionLabel) => {
  setLoading(true);
  try {
    // 提交异步任务，获取task_id
    const res = await axios.post("/api/chat/send", { chatId, userInput, emotionLabel });
    const { task_id } = res.data.data;
    // 轮询获取结果，间隔500ms
    const interval = setInterval(async () => {
      const resultRes = await axios.get("/api/chat/task/result", { params: { task_id } });
      if (resultRes.data.code === 200) {
        clearInterval(interval);
        setMessageList(prev => [...prev, { role: "assistant", content: resultRes.data.data.result.response }]);
        setLoading(false);
      }
    }, 500);
  } catch (err) {
    setLoading(false);
    // 捕获429/500异常，显示儿童友好提示
    setMessageList(prev => [...prev, { role: "assistant", content: err.response.data.msg }]);
  }
};
```

- 动态调整worker并发数/副本数，适配业务流量变化；
- 手动重试失败任务，排查LLM API调用问题。
